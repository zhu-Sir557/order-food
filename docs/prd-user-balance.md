# 产品需求文档（PRD）：用户注册登录 + 余额支付系统

## 项目信息

| 项目 | 内容 |
|------|------|
| 项目名称 | order_food |
| 编程语言/技术栈 | 后端：Java 21 + Spring Boot 3.2.5 + MyBatis-Plus 3.5.7 + MySQL 8.0；后台管理前端：Vue 3 + Element Plus + Vite（端口5173）；H5点餐前端：Vue 3 + Vant 4 + Vite + Pinia + TypeScript（端口5174） |
| 文档版本 | v1.0 |
| 创建日期 | 2025-07-11 |
| 迭代范围 | 用户注册登录 + 滑块验证 + 点卡批量创建/兑换 + 余额支付 |

### 原始需求复述

在现有点餐系统基础上新增两个核心模块：

1. **用户登录注册功能**：H5端用户可注册账户（账户名唯一），通过账户名+密码+滑块图片验证登录，登录后可查看自己的订单、余额等信息。
2. **余额支付系统**：后台管理员可批量创建点卡（指定额度、张数）并发放给指定用户；H5端用户可用卡号卡密兑换点卡余额到账户，并使用账户余额进行订单支付。

### 现有系统背景

| 维度 | 现状 |
|------|------|
| 用户体系 | 仅有临时用户（TempUser：id/token/createTime/deleted），扫码即用，无注册登录。H5端通过 `POST /api/h5/user/temp` 自动创建临时用户获取JWT token（role=CUSTOMER） |
| 订单关联 | OrderInfo.tempUserId 关联临时用户 |
| 支付体系 | 订单支付仅修改状态（PENDING_PAY → PENDING_ACCEPT），无支付方式字段，无实际扣款逻辑 |
| 认证拦截 | CustomerAuthInterceptor 仅拦截 `/api/h5/orders/**`，从JWT中提取 tempUserId 存入 request attribute |
| H5用户Store | 管理 token/tempUserId/tableId/tableCode/tableName，App启动时自动创建临时用户 |
| 后台菜单 | 仪表盘、菜品管理、分类管理、订单管理、桌台管理、轮播图管理、用户管理（mock数据） |
| 数据库表 | admin_user, temp_user, category, dish, dining_table, order_info, order_item, banner |

## 一、产品目标

| # | 目标 | 说明 |
|---|------|------|
| G1 | **建立注册用户体系** | 在现有临时用户基础上新增注册会员（Member）体系，支持账户名+密码+滑块验证登录，使顾客拥有可跨设备使用的持久身份，能查看个人订单和余额 |
| G2 | **实现点卡余额充值闭环** | 后台批量生成点卡 → 发放/用户自助兑换 → 余额入账，形成完整的预充值资金流，为餐厅提供预收款能力 |
| G3 | **打通余额支付链路** | 订单支付接入余额扣款，实现从充值到消费的完整闭环，确保扣款原子性、余额变动可追溯 |

## 二、用户故事

| # | 角色 | 用户故事 |
|---|------|----------|
| US1 | H5顾客 | 作为一名顾客，我想注册一个账户（账户名+密码），以便拥有持久身份并使用余额支付功能 |
| US2 | H5顾客 | 作为一名顾客，我想通过账户名+密码+滑块验证登录，以便安全地访问我的订单和余额 |
| US3 | H5顾客 | 作为一名顾客，我想输入卡号和卡密兑换点卡，以便将点卡余额充入我的账户 |
| US4 | H5顾客 | 作为一名顾客，我想在下单时用账户余额支付，以便快速完成订单而无需第三方支付 |
| US5 | 后台管理员 | 作为一名管理员，我想批量创建点卡（指定额度和张数），以便生成一批可发放的充值卡 |

## 三、需求池（P0/P1/P2 分级）

### P0 — 必须有（MVP 核心）

| 编号 | 模块 | 需求 | 验收标准 |
|------|------|------|----------|
| P0-01 | H5-注册 | 顾客在H5端注册账户，输入账户名（唯一）、密码、确认密码 | 账户名重复时提示"账户名已存在"；密码长度≥6位；注册成功后自动登录并返回JWT |
| P0-02 | H5-登录 | 顾客通过账户名+密码+滑块图片验证登录 | 滑块未通过时禁止提交；账户名/密码错误时提示；登录成功返回JWT并存储到localStorage |
| P0-03 | H5-滑块验证 | 登录页集成滑块图片验证组件 | 后端生成验证图（含缺口图+背景图+滑块图）；前端拖拽滑块拼合缺口；后端校验偏移量是否在容差范围内；验证结果一次性有效 |
| P0-04 | 后台-点卡批量创建 | 管理员指定额度和张数，批量生成点卡 | 每张点卡生成唯一卡号+卡密；生成后展示卡号卡密列表，支持导出；点卡初始状态为"未使用" |
| P0-05 | H5-点卡兑换 | 顾客输入卡号+卡密，将点卡余额充入账户 | 卡号/卡密错误时提示；已使用的卡提示"卡已兑换"；兑换成功后余额实时增加；卡状态变为"已使用"并记录使用者 |
| P0-06 | H5-余额支付 | 顾客下单后可用账户余额支付订单 | 余额不足时提示"余额不足，请充值"并禁用支付按钮；余额充足时扣款成功，订单状态变为"待接单"；扣款与状态变更在同一事务内 |
| P0-07 | 后端-会员认证 | 注册/登录接口开放，余额相关接口需会员身份认证 | JWT中包含role=MEMBER标识；MemberAuthInterceptor校验会员身份；未登录访问余额接口返回401 |

### P1 — 应该有（提升体验）

| 编号 | 模块 | 需求 | 验收标准 |
|------|------|------|----------|
| P1-01 | 后台-点卡发放 | 管理员可将点卡指定发放给某个注册用户 | 选择用户+选择点卡 → 发放后点卡状态标记为"已发放"，记录发放对象；用户端可见已发放给本人的卡 |
| P1-02 | H5-余额变动记录 | 顾客可查看余额变动明细（充值/消费） | 列表展示变动类型、金额、变动后余额、时间；支持按类型筛选 |
| P1-03 | 后台-用户余额管理 | 管理员可查看注册用户列表及其余额 | 用户列表展示账户名、余额、注册时间、订单数；可查看单用户余额变动记录 |
| P1-04 | 后台-点卡列表管理 | 管理员可查看所有点卡，按状态/批次筛选 | 展示卡号、额度、状态（未使用/已使用/已发放）、批次号、创建时间；支持导出 |
| P1-05 | H5-我的页面改造 | "我的"页面展示登录状态、账户名、余额 | 未登录时显示"点击登录"入口；已登录时显示账户名+余额+退出按钮；新增"兑换点卡"和"余额记录"入口 |

### P2 — 可以有（锦上添花）

| 编号 | 模块 | 需求 | 验收标准 |
|------|------|------|----------|
| P2-01 | 后台-点卡批次管理 | 管理员可管理点卡批次（按批次查看/统计） | 批次列表展示总额度、已兑换数量、未兑换数量；支持按批次导出 |
| P2-02 | 后台-数据统计扩展 | 仪表盘增加充值总额、点卡兑换率等指标 | 统计充值总额、兑换率、余额消费占比 |
| P2-03 | H5-密码修改 | 顾客可修改登录密码 | 需验证旧密码后设置新密码 |
| P2-04 | 后端-点卡过期机制 | 点卡可设置有效期，过期自动作废 | 创建点卡时可指定过期时间；兑换时校验是否过期 |

**交互要素**：

- 滑块校验失败时，自动刷新验证图，提示"验证失败，请重试"
- 登录成功后跳转回来源页（或首页），JWT存入localStorage

#### 4.3 滑块验证组件（通用）

```
┌─────────────────────────────┐
│  ┌───────────────────────┐  │
│  │  [背景图 with 缺口]    │  │  ← 300x150 背景图，随机位置有缺口
│  │        ▓▓▓             │  │
│  └───────────────────────┘  │
│  ┌───────────────────────┐  │
│  │ [滑块]  →→→  拖动滑块  │  │  ← 滑块轨道，拖拽滑块块至缺口位置
│  └───────────────────────┘  │
└─────────────────────────────┘
```

**交互要素**：
- 组件加载时调用 `GET /api/h5/captcha/slider` 获取背景图、缺口图、滑块图、captchaId
- 用户拖拽滑块，松手时调用 `POST /api/h5/captcha/slider/check` 提交 captchaId + 偏移量X
- 校验通过：触发 success 事件，父组件启用提交按钮
- 校验失败：自动刷新验证图，触发 fail 事件
- 容差范围：±5px

## 五、数据模型设计建议

> 以下为产品层面的数据模型建议，最终实现以架构师设计为准。

### 新增表

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| `member` | 注册会员表 | id, username(唯一), password(BCrypt), balance(DECIMAL), phone, status, create_time, update_time, deleted |
| `recharge_card` | 点卡表 | id, card_no(唯一), card_secret, amount, status(0未使用/1已发放/2已使用), batch_no, assigned_member_id, used_member_id, used_time, create_time, deleted |
| `balance_record` | 余额变动记录表 | id, member_id, type(RECHARGE/PAY), amount, balance_after, related_order_id, related_card_id, remark, create_time |

### 修改表

| 表名 | 变更 | 说明 |
|------|------|------|
| `order_info` | 新增 `member_id BIGINT DEFAULT NULL` | 关联注册会员，未登录时为NULL（仍用temp_user_id） |
| `order_info` | 新增 `pay_method TINYINT DEFAULT NULL` | 支付方式：1余额支付 2微信 3支付宝（NULL表示未支付） |

### 认证体系

| 身份类型 | JWT role | 拦截器 | 说明 |
|----------|----------|--------|------|
| 管理员 | ADMIN | AdminAuthInterceptor | 拦截 `/api/admin/**` |
| 临时用户 | CUSTOMER | CustomerAuthInterceptor | 拦截 `/api/h5/orders/**`，保持现有行为 |
| 注册会员 | MEMBER | MemberAuthInterceptor（新增） | 拦截 `/api/h5/member/**`，校验role=MEMBER，存memberId到request |

> **临时用户与注册会员共存策略**：用户未登录时为临时用户（现有流程不变），登录后变为注册会员。注册会员的订单关联 member_id，临时用户的订单仍关联 temp_user_id。余额支付仅对注册会员开放。

## 七、待确认问题

| # | 问题 | 影响 | 建议默认值 |
|---|------|------|-----------|
| Q1 | 临时用户与注册会员的关系？注册后是否需要绑定已有临时用户的订单？ | 影响数据模型和订单归属 | 建议不绑定，注册后新订单关联member_id，历史临时订单不迁移。注册会员和临时用户共存 |
| Q2 | 滑块验证码的实现方式？后端生成图片还是使用第三方库？ | 影响技术选型 | 建议后端用Java图片处理（如easy-captcha或自行实现）生成滑块图，不依赖第三方服务 |
| Q3 | 点卡卡号和卡密的格式规则？ | 影响点卡生成逻辑 | 建议卡号格式：RC+yyyyMMdd+6位随机数（如RC20250711001234），卡密：16位随机字母数字混合 |
| Q4 | 余额支付失败后是否需要回滚？例如库存已扣但余额扣款失败？ | 影响事务设计 | 建议余额扣款与订单状态变更在同一事务内，任一步骤失败则整体回滚 |
| Q5 | 点卡是否支持部分兑换？例如100元卡只兑换50元？ | 影响兑换逻辑 | 建议不支持，一张卡一次性全额兑换到账户余额 |
| Q6 | 注册是否需要手机号/短信验证？ | 影响注册流程复杂度 | MVP不需要，仅账户名+密码即可。手机号注册列为P2 |
| Q7 | 余额是否支持提现？ | 影响资金管理 | 建议不支持提现，余额仅用于消费 |
| Q8 | 微信支付/支付宝是否需要真实接入？ | 影响支付模块开发量 | 本次仅实现余额支付，微信/支付宝保持现有模拟逻辑不变 |
| Q9 | 已登录会员的订单是否仍需要关联临时用户？ | 影响订单数据模型 | 建议已登录会员下单时member_id和temp_user_id都填写（temp_user_id为当前会话的临时用户ID），便于数据追溯 |
| Q10 | 点卡发放后用户是否需要在H5端看到"待领取"的卡？还是直接自动充值？ | 影响发放与兑换流程 | 建议发放后用户在H5端"待领取点卡"列表中可见，需用户主动确认领取（兑换）。也可支持自动兑换 |
