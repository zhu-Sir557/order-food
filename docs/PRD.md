# 产品需求文档（PRD）：智能点餐系统

## 项目信息

| 项目 | 内容 |
|------|------|
| 项目名称 | order_food |
| 编程语言/技术栈 | 后端：Java 21 + Spring Boot + MySQL 8.0；前端：Vue（后台管理 + H5点餐） |
| 文档版本 | v1.0 |
| 创建日期 | 2025-07-11 |

### 原始需求复述

用户需要开发一个完整的点餐系统，包括：
1. **后台管理系统**：餐厅管理员使用，管理菜品、分类、订单、用户等
2. **前端H5风格点餐界面**：顾客使用，手机端浏览菜品、下单、支付

技术栈要求：后端 Java 21 + Spring Boot + MySQL 8.0，前端 Vue（后台管理 + H5点餐）。

---

## 一、产品目标

| # | 目标 | 说明 |
|---|------|------|
| G1 | **提升点餐效率** | 顾客通过 H5 自助浏览菜单、下单、支付，减少人工记录环节，降低服务员工作量，缩短从入座到下单的平均时间 |
| G2 | **集中化餐厅运营管理** | 为餐厅管理员提供统一的菜品、分类、订单、桌台管理后台，实现数据驱动运营决策 |
| G3 | **保障订单链路可靠性** | 确保从顾客下单到后厨接单、订单状态流转的全链路可追踪、状态一致，避免漏单错单 |

## 二、用户故事

| # | 角色 | 用户故事 |
|---|------|----------|
| US1 | 顾客 | 作为一名顾客，我想在手机上浏览菜单并查看菜品图片和价格，以便快速决定要点什么 |
| US2 | 顾客 | 作为一名顾客，我想将菜品加入购物车并一键下单，以便高效完成点餐流程 |
| US3 | 顾客 | 作为一名顾客，我想实时查看我的订单状态（待接单/制作中/已完成），以便知道何时可以取餐 |
| US4 | 餐厅管理员 | 作为一名餐厅管理员，我想在后台管理菜品上下架和分类，以便灵活调整菜单 |
| US5 | 餐厅管理员 | 作为一名餐厅管理员，我想查看和管理所有订单及桌台状态，以便协调堂食运营 |
| US6 | 后厨人员 | 作为后厨人员，我想按时间顺序查看新订单并标记制作进度，以便有序完成菜品制作 |

## 三、需求池（P0/P1/P2 分级）

### P0 — 必须有（MVP 核心）

| 编号 | 模块 | 需求 | 验收标准 |
|------|------|------|----------|
| P0-01 | H5点餐-菜品浏览 | 顾客可在 H5 端按分类浏览菜品，查看名称、图片、价格、描述 | 分类切换流畅；菜品信息完整展示 |
| P0-02 | H5点餐-购物车 | 顾客可将菜品加入购物车，调整数量，查看合计金额 | 加购/减数量/清空操作正常；金额实时计算 |
| P0-03 | H5点餐-下单 | 顾客确认购物车后提交订单，选择堂食桌号或备注 | 下单成功生成订单号；库存不足时有提示 |
| P0-04 | H5点餐-支付（模拟） | 提供模拟支付页面，点击支付后订单状态变为"已支付/待接单" | 支付成功后订单状态正确流转 |
| P0-05 | H5点餐-订单查看 | 顾客可查看自己的历史订单及当前订单状态 | 订单列表按时间倒序；状态实时反映 |
| P0-06 | 后台-菜品管理 | 管理员可对菜品进行增删改查（CRUD），包括图片上传、上下架 | CRUD 功能完整；下架菜品不在 H5 显示 |
| P0-07 | 后台-分类管理 | 管理员可对菜品分类进行增删改查，支持排序 | 分类 CRUD 完整；支持拖拽或数值排序 |
| P0-08 | 后台-订单管理 | 管理员可查看所有订单列表，筛选状态，查看详情，手动改状态 | 支持按状态/时间/桌号筛选；可接单/完成订单 |
| P0-09 | 后台-桌台管理 | 管理员可管理堂食桌台（编号、容量、状态） | 桌台 CRUD；状态：空闲/就餐中 |
| P0-10 | 后端-订单状态机 | 订单状态流转：待支付 → 已支付/待接单 → 制作中 → 已完成 → 已取餐 | 状态只能按定义顺序流转；非法流转被拒绝 |
| P0-11 | 后端-API 权限 | 后台 API 需管理员鉴权；H5 API 需顾客身份（桌号或临时用户） | 未授权请求返回 401；越权操作返回 403 |

### P1 — 应该有（提升体验）

| 编号 | 模块 | 需求 | 验收标准 |
|------|------|------|----------|
| P1-01 | 后台-用户管理 | 管理员可管理顾客用户信息，查看消费记录 | 用户列表可搜索；可查看单用户订单汇总 |
| P1-02 | 后台-数据统计 | 仪表盘展示今日订单数、营业额、热销菜品 Top10 | 数据实时统计；图表可视化展示 |
| P1-03 | H5点餐-菜品搜索 | 顾客可通过关键词搜索菜品 | 支持模糊匹配菜品名称 |
| P1-04 | H5点餐-口味/规格选择 | 顾客下单时可选择口味（辣度）或规格（大/小份） | 规格选项影响价格；订单详情中体现 |
| P1-05 | 后台-订单通知 | 新订单到达时后台有声音/弹窗提醒 | 实时推送；管理员可确认接单 |
| P1-06 | 后端-菜品库存 | 菜品支持设置库存，售罄自动下架或标记 | 库存为 0 时 H5 显示"已售完"；下单扣减库存 |

### P2 — 可以有（锦上添花）

| 编号 | 模块 | 需求 | 验收标准 |
|------|------|------|----------|
| P2-01 | H5点餐-菜品评价 | 顾客完成订单后可对菜品打分评价 | 评分 1-5 星；评价在菜品详情展示 |
| P2-02 | 后台-会员/优惠券 | 支持优惠券发放与核销 | 优惠券 CRUD；下单时可选用 |
| P2-03 | 后台-多门店 | 支持多门店管理，数据隔离 | 不同门店菜品/订单独立 |
| P2-04 | H5点餐-扫码点餐 | 顾客扫桌台二维码直接进入点餐页面，自动绑定桌号 | 扫码后自动选桌；无需手动输入 |
| P2-05 | 后端-营业报表导出 | 管理员可导出日/周/月营业报表（Excel） | 报表含订单明细、营业额、菜品销量 |

## 五、待确认问题

| # | 问题 | 影响 | 建议 |
|---|------|------|------|
| Q1 | 就餐模式是否仅支持堂食？是否需要外卖/自取？ | 影响 P0-03 下单流程设计 | MVP 先做堂食，外卖列为 P2 |

| Q7 | H5 端是否需要部署为独立应用，还是嵌入微信小程序/公众号？ | 影响前端打包和部署方案 | 建议 H5 独立部署，兼容微信内置浏览器 |
| Q8 | 数据统计的粒度？是否需要按时段（午餐/晚餐）分析？ | 影响统计模块设计 | MVP 做日维度统计，时段分析列为 P2 |

## 附录：订单状态流转图

```
┌─────────┐     支付成功     ┌──────────┐    管理员接单   ┌─────────┐
│ 待支付   │ ──────────────→ │ 待接单    │ ────────────→ │ 制作中   │
└─────────┘                  └──────────┘               └─────────┘
                                  │                          │
                                  │ 超时取消                  │ 制作完成
                                  ▼                          ▼
                             ┌─────────┐              ┌──────────┐
                             │ 已取消   │              │ 已完成    │
                             └─────────┘              └──────────┘
                                                           │
                                                           │ 顾客取餐
                                                           ▼
                                                      ┌──────────┐
                                                      │ 已取餐    │
                                                      └──────────┘
```

**状态说明**：
- 待支付：顾客已提交订单但未支付
- 待接单：支付完成，等待后厨/管理员确认
- 制作中：后厨已接单，正在制作
- 已完成：菜品制作完成，等待取餐
- 已取餐：顾客已取餐，订单结束
- 已取消：超时未支付或管理员手动取消
