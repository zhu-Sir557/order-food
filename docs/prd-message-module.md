# 产品需求文档（PRD）：消息模块（增量规划）

> 本文档为 **order_food 智能点餐系统** 的消息模块**增量规划 PRD**，属于规划阶段，仅定义产品需求，**不含任何实现代码**。

## 项目信息

| 项目 | 内容 |
|------|------|
| 项目名称 | order_food（消息模块增量） |
| 文档版本 | v0.1（规划稿） |
| 创建人 | 许清楚（产品经理） |
| 创建日期 | 2025-07-15 |
| 关联系统 | 后台管理（admin，Vue3 + Element Plus）；H5 点餐端（Vue3 + Vite + TS） |
| 后端现状 | Spring Boot 3.2.5 / Java 21 / MyBatis-Plus / Redis / MySQL；统一响应 `Result<T>{code,message,data}`，成功 code=200 |
| 增量约束 | 当前**无** WebSocket/STOMP/SockJS 代码、**无** message 相关表、**无**消息 UI；需全新引入；新增接口只需把路径注册进现有拦截器，无需新写拦截器 |

### 原始需求复述（来自交付总监转述）

> 管理员可通过后台向用户端（H5）发送消息，用户端通过 WebSocket 实时接收；消息类型需定义清楚：包含**全员发送**与**指定用户发送**。

## 一、产品目标

| # | 目标 | 说明 |
|---|------|------|
| G1 | **建立管理员→用户实时触达通道** | 餐厅管理员可主动向 H5 用户（会员/游客）推送通知，替代"只能等用户来"的被动模式 |
| G2 | **让消息"即时可见、可回溯"** | H5 用户实时收到消息并有消息中心可查看历史、标记已读，不遗漏重要通知 |
| G3 | **以最小侵入接入现有架构** | 复用现有鉴权（token / 拦截器）、OSS 上传与 Result 响应体系，仅增量引入 WebSocket 与消息表，不改动既有业务逻辑 |

## 三、消息类型定义（重点）

消息类型由枚举 `MessageType` 定义，发送时必选其一。范围（接收人）由 `receiverScope` 与具体接收人集合共同决定。

### 3.1 核心类型（P0 必须支持）

| 类型 | 枚举值 | 语义 | 接收方范围定义 | 发送入口 |
|------|--------|------|----------------|----------|
| 全员广播 | `BROADCAST` | 管理员向全体 H5 用户发送一条统一消息 | 范围 = 所有当前具有 H5 身份的用户。**待确认**：是否包含扫码临时游客（temp_user）？见第六节 Q1（推荐：包含，在线即可达） | 后台"消息推送"页，类型选"全员广播"，无需选人 |
| 指定用户 | `SPECIFIED` | 管理员向选定的一个或多个用户定向发送 | 范围 = 选定的用户 ID 列表。**待确认**：是否含临时游客？见 Q1（推荐 P0 仅会员 member，临时游客身份易逝难稳定寻址） | 后台"消息推送"页，类型选"指定用户"，从会员列表多选 |

### 3.2 扩展类型（P1/P2，用户未要求，按需裁剪）

| 类型 | 枚举值 | 语义 | 建议优先级 | 说明 |
|------|--------|------|-----------|------|
| 系统通知 | `SYSTEM` | 由系统事件自动触发（订单状态变更、余额变动等），非管理员手动发送 | P2 | 属自动化触达，需事件总线，建议作为后续迭代，不与手动推送混用同一发送页 |

> 说明：本 PRD 以 `BROADCAST` 与 `SPECIFIED` 为交付核心，扩展类型仅占位，避免喧宾夺主。

### 3.3 消息内容形式

| 内容要素 | 是否必填 | 形式 | 说明 |
|----------|----------|------|------|
| 标题 `title` | 必填 | 纯文本（≤50 字） | 列表与通知摘要展示 |
| 正文 `content` | 必填 | 纯文本（P0）；富文本（P1 可选） | P0 仅纯文本，降低前端复杂度 |
| 图片 `imageUrl` | 可选 | OSS URL | 复用现有 `/api/admin/upload` 上传，返回 OSS 地址；P0 即支持 |
| 跳转链接 `linkUrl` | 可选 | 合法 URL（站内页如菜品/活动页） | 点击消息跳转到目标页；需做基础 URL 合法性校验 |
| 类型 `type` | 必填 | 枚举 | BROADCAST / SPECIFIED（/ SYSTEM） |

## 四、需求池（P0 / P1 / P2）

### P0 — 必须有（MVP 核心）

| 编号 | 模块 | 需求（做什么） | 验收标准 |
|------|------|----------------|----------|
| P0-01 | 后台-发送页 | 提供"消息推送"页：选择类型（全员/指定）、指定时多选会员、编辑标题+正文、可选上传图片、可选填跳转链接、点击发送 | 表单校验：标题/正文必填；指定类型必须至少选 1 个接收人；发送成功返回消息记录 ID |
| P0-02 | 后台-发送记录 | 列表展示历史发送（时间/类型/范围/标题/状态），支持查看详情 | 列表按时间倒序；状态含"已发送/已撤回"；分页正常 |
| P0-03 | 后端-消息表 | 新建 `message`（消息主表：id/type/receiver_scope/sender_id/title/content/image_url/link_url/status/create_time）+ 已读维度表 `message_read`（message_id/user_id/read_time）或消息行内带 receiver 维度 | 表结构符合现有 schema 维护习惯；可经 schema.sql 或独立 migration 脚本入库 |
| P0-04 | 后端-发送接口 | 新增 `/api/admin/message/send`：创建消息记录 + 按范围解析接收人 | 返回 `Result<MessageVO>`；参数非法返回 PARAM_ERROR；非 admin 调用被 AdminAuthInterceptor 拦截（401） |
| P0-05 | 后端-WebSocket | 引入 `spring-boot-starter-websocket` + STOMP/SockJS，建立 `/ws` 端点；H5 连接时携带 `Authorization: Bearer <token>` 完成鉴权并订阅个人队列 | 单实例可用应用内 `SimpMessagingTemplate` 广播；Nginx 需配置 Upgrade 头转发（列为部署项） |
| P0-06 | 后端-实时推送 | 发送成功后，通过 WebSocket 将消息实时推送到目标用户（全员→广播通道；指定→对应用户会话） | 在线用户毫秒级收到；不在线用户落库（见 P1-01 离线拉取可作为 P0 兜底） |
| P0-07 | H5-连接与接收 | H5 封装 WebSocket 客户端，复用现有 token 自动鉴权；监听个人消息通道 | 连接断开可自动重连；401 跳登录逻辑与现有一致 |
| P0-08 | H5-消息中心 | 新增"消息中心"页（入口挂在 `/me` 我的页）：列表展示消息（标题/摘要/时间/未读红点），点击进入详情 | 列表按时间倒序；未读项有红点；点击标记已读并写 `message_read` |
| P0-09 | H5-消息详情 | 消息详情页展示标题、正文、图片（如有）、跳转链接（如有） | 点击跳转链接安全打开目标页；已读状态正确更新 |

### P1 — 应该有（体验提升）

| 编号 | 模块 | 需求 | 验收标准 |
|------|------|------|----------|
| P1-01 | 后端-离线消息 | WebSocket 不在线时消息仅落库；用户上线/打开消息中心时拉取未读消息 | 离线期间消息不丢失；进入消息中心即补齐未读列表 |
| P1-02 | H5-未读聚合 | 消息中心标题/入口显示未读总数红点 | 红点数字实时反映未读数；全部已读后清零 |
| P1-03 | H5-实时 Toast | 收到新消息时 H5 顶部弹 Toast 提醒（非阻断） | 常驻连接下即时弹出；点击 Toast 跳转详情；可手动关闭 |
| P1-04 | 后台-撤回 | 管理员对"已发送"消息在限定时间（建议 5 分钟）内撤回 | 撤回后在线用户消息消失/标记"已撤回"；超时不可撤回；仅发送者/管理员可操作 |
| P1-05 | 后台-模板/分类 | 常用消息模板与消息分类标签 | 可保存模板快速填充；列表可按分类筛选 |
| P1-06 | 内容-富文本 | 正文支持图文混排富文本 | 前端富文本编辑器可编辑；后端安全存储并做 XSS 过滤 |

### P2 — 可以有（锦上添花）

| 编号 | 模块 | 需求 | 验收标准 |
|------|------|------|----------|
| P2-01 | 系统-SYSTEM 通知 | 订单状态变更/余额变动等事件自动生成 SYSTEM 消息并推送 | 事件触发即生成；与手动推送共用接收与展示通道 |
| P2-02 | 部署-多实例 | 多实例部署时改用 Redis pub/sub 做消息广播 | 单实例升级多实例后推送不丢；当前单实例可暂缓 |
| P2-03 | 后台-触达统计 | 消息送达率/已读率统计看板 | 展示每条消息已读人数占比 |

## 六、待确认问题（需用户拍板）

| # | 待确认决策点 | 推荐选项（Recommended） | 理由 |
|---|--------------|--------------------------|------|
| Q1 | 指定用户 / 全员广播的接收范围是否包含**扫码临时游客（temp_user）**？ | **P0 包含全员广播（在线即可达）；指定用户 P0 仅会员(member)** | temp_user 身份存在于桌台扫码会话内，广播按在线会话可达；但 temp_user ID 易逝、难稳定多选寻址，指定发送先限会员更稳妥 |
| Q2 | 消息是否支持**图片 / 富文本**？ | **P0 支持图片（OSS 复用）；富文本 P1** | 图片需求用户已明确；富文本编辑器引入复杂度高，先纯文本+图片满足核心 |
| Q3 | 是否需要**离线消息落库 + 未读聚合**？ | **P0 必须落库（离线不丢，上线/进中心拉取）；未读红点聚合 P1** | 落库是可靠触达基础能力，应进 MVP；红点聚合为体验增强放 P1 |
| Q4 | 是否支持**撤回**？ | **P1（发送后 5 分钟内可撤回）** | 纠错有价值但非首版必需；限定时间窗避免历史消息被篡改 |
| Q5 | 是否需要**实时 Toast 弹窗**提醒？ | **P1（常驻 WebSocket 连接下弹出）** | 显著提升触达，但依赖 H5 长连接稳定性，先以消息中心列表保证可达，Toast 作为增强 |
| Q6 | WebSocket 鉴权方式 | **复用现有 H5 token（Bearer）+ 现有拦截/解析逻辑** | 不新增鉴权体系，最小化改动；连接握手时校验 token 并绑定用户会话 |

## 七、技术接入说明（供架构师参考，非实现）

1. **WebSocket 引入**：后端新增 `spring-boot-starter-websocket` 依赖与 `WebSocketConfig`（STOMP + SockJS，`/ws` 端点）；单实例用 `SimpMessagingTemplate` 应用内广播，多实例再上 Redis pub/sub（P2）。
2. **鉴权复用**：H5 WebSocket 握手携带 `Authorization: Bearer <token>`，复用现有 JWT 解析确定 memberId/tempUserId，绑定到 WebSocket session；不新增拦截器。
3. **接口注册**：新增 admin 消息接口路径（`/api/admin/message/**`）注册进 `AdminAuthInterceptor`；H5 消息接口（`/api/h5/message/**`）复用现有鉴权注册方式，在 `WebMvcConfig` 追加路径即可。
4. **数据落库**：按现有 schema 维护习惯，新增 `message` / `message_read`（或 receiver 维度）表，补 schema 或独立 migration 脚本。
5. **图片复用**：消息图片直接复用 `/api/admin/upload` 返回的 OSS URL。
6. **部署**：Nginx 需增加 `Upgrade` / `Connection` 头转发到 8080，保证 WebSocket 握手通过。
7. **测试**：后端测试沿用 JUnit5 + MockitoExtension + standalone MockMvc 模式，新增 `MessageControllerTest` / `MessageServiceTest`（当前 CI `mvn -DskipTests package` 跳过测试，新增测试不强制门禁）。
